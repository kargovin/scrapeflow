## 2. Database Schema Changes

### Migration Workflow — Alembic Autogenerate

This project uses `alembic revision --autogenerate`. **Do not write raw SQL migrations by hand.** The correct workflow for each migration is:

1. Update the SQLAlchemy model(s) in `api/app/models/` to reflect the target schema
2. Run `alembic revision --autogenerate -m "description"` — Alembic diffs the models against the live DB and generates `op.*` Python DSL code
3. Review the generated file — autogenerate is not perfect, check for missed indexes or constraints
4. Append the two things autogenerate cannot produce (see below)
5. Run `alembic upgrade head`

**The raw SQL in each section below is the intended target schema — treat it as a reference, not as migration code.** An engineer implementing these migrations writes SQLAlchemy models matching the schema, then autogenerates.

**Two things autogenerate cannot handle — must be appended manually:**

**1. The data migration in §2.3** — copying existing job outcomes into `job_runs` is data manipulation, not DDL. Append inside `upgrade()` after `op.create_table('job_runs', ...)`:
```python
op.execute("""
    INSERT INTO job_runs (id, job_id, status, result_path, error, completed_at, created_at)
    SELECT gen_random_uuid(), id, status, result_path, error, updated_at, created_at
    FROM jobs WHERE status != 'pending'
""")
```

**2. `ALTER TYPE jobstatus ADD VALUE 'processing'` in §2.6** — Alembic autogenerate does not detect ENUM value additions. This must be its own separate migration file with `transaction = False` at the module level:
```python
# Cannot run inside a transaction — set at module level, not inside upgrade()
transaction = False

def upgrade() -> None:
    op.execute("ALTER TYPE jobstatus ADD VALUE 'processing' AFTER 'running'")

def downgrade() -> None:
    pass  # Postgres has no ALTER TYPE DROP VALUE — downgrade is a no-op
```

**Practical result:** Phase 2 needs **two `alembic revision` runs**: one autogenerated revision covering all structural changes (new tables, new columns, dropped columns, indexes), with the data migration appended manually; and one hand-written revision for the ENUM value addition. This matches how Phase 1 was built.

### 2.3 Migration: Create `job_runs` table

```sql
CREATE TABLE job_runs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    job_id          UUID NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
    status          VARCHAR(20) NOT NULL,
    CONSTRAINT job_runs_status_check CHECK (
        status IN ('pending', 'running', 'processing', 'completed', 'failed', 'cancelled')
    ),
    result_path     TEXT NULL,
    diff_detected   BOOLEAN NULL,
    diff_summary    JSONB NULL,
    error           TEXT NULL,
    started_at      TIMESTAMPTZ NULL,
    completed_at    TIMESTAMPTZ NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_job_runs_job_id ON job_runs (job_id);
CREATE INDEX idx_job_runs_status ON job_runs (status);
CREATE INDEX idx_job_runs_created_at ON job_runs (created_at);
```

**Note:** `cancelled` is included in the CHECK constraint because `DELETE /jobs/{id}` writes `job_runs.status = 'cancelled'` (see §5.5). The BRIN index on `created_at` has been removed — it is redundant alongside the B-tree (see Issue 20 fix).

**`result_path` convention — always stores the `history/` path:**

Workers write two MinIO paths per run:
- `latest/{job_id}.{ext}` — always overwritten, reflects current state, used by webhook payloads
- `history/{job_id}/{timestamp}.{ext}` — append-only, one object per run, used by diff algorithm

`job_runs.result_path` stores the **`history/` path** — the immutable, per-run result. This is the only path that lets the diff algorithm retrieve two specific consecutive runs by querying `job_runs.result_path` on the current and previous run rows. The `latest/` path is always derivable as `latest/{job_id}.{ext}` and does not need to be stored.

The result message `minio_path` field (§3.3) carries the `history/` path. Workers must write to `latest/` for convenience access but publish the `history/` path in the result event.

**Data migration — copy existing job outcomes into job_runs:**
```sql

SELECT
    gen_random_uuid(),
    id,
    status,
    result_path,
    error,
    updated_at,

```

### 2.8 Migration: Add `nats_stream_seq` to `job_runs`

Required by §6.3 (MaxDeliver advisory subscriber). The advisory message from NATS contains only the stream sequence number — no `job_id` or `run_id`. This column is the bridge that lets the advisory handler identify which `job_runs` row to mark failed.

```sql
ALTER TABLE job_runs ADD COLUMN nats_stream_seq BIGINT NULL;

-- Sparse index — only on rows where the value is set (most rows are NULL until a worker picks up the job)
CREATE INDEX idx_job_runs_nats_stream_seq
    ON job_runs (nats_stream_seq)
    WHERE nats_stream_seq IS NOT NULL;
```

**Who sets it, and when:** The result consumer sets `nats_stream_seq` when it receives the first `status: "running"` message from a worker. At that point the worker has already read `msg.Metadata().Sequence.Stream` and included it in the result payload as `nats_stream_seq`. The result consumer writes it to `job_runs.nats_stream_seq` alongside `started_at`.

See §3.3 — the job result message schema includes `nats_stream_seq` (present only on `status: "running"` messages).

### 3.1 NATS Stream Change

**Before (Phase 1):**
```
Stream: SCRAPEFLOW
Subjects: scrapeflow.jobs.run, scrapeflow.jobs.result
Retention: WorkQueuePolicy
```

**After (Phase 2):**
```
Stream: SCRAPEFLOW
Subjects: scrapeflow.jobs.>          ← wildcard replaces explicit list
Retention: WorkQueuePolicy
```

The `>` wildcard matches all subjects with one or more tokens after `scrapeflow.jobs.`. This covers all current and future subjects without stream reconfiguration.

**Migration — dev environment (current state):** `docker compose down -v` wipes all volumes (NATS, Postgres, MinIO) and recreates from scratch. Acceptable now because the project is dev-only with no persistent data worth keeping.

**Migration — staging/production (before those environments exist):** Replace `down -v` with an in-place stream edit using the NATS CLI:
```bash
nats stream edit SCRAPEFLOW --subjects "scrapeflow.jobs.>" --server nats:4222
```
If the NATS version doesn't support `stream edit` for subject changes, delete and recreate the stream without touching other volumes:
```bash
nats stream rm SCRAPEFLOW --force --server nats:4222
nats stream add SCRAPEFLOW --subjects "scrapeflow.jobs.>" --retention work \
  --max-deliver 3 --storage file --replicas 1 --server nats:4222
```

**Why the full fix is deferred:** The project currently has no staging environment and no persistent dev data. The risk `down -v` poses (data loss) does not exist yet. This must be resolved before staging is provisioned — add it to the staging setup checklist.

### 3.3 Updated Message Schemas

**Job dispatch — HTTP / Playwright (API → worker):**
```json
{
  "job_id": "uuid",
  "run_id": "uuid",
  "url": "https://example.com",
  "output_format": "html|markdown|json",
  "playwright_options": {
    "wait_strategy": "load",
    "timeout_seconds": 60,
    "block_images": false
  }
}
```
`playwright_options` is present only in Playwright messages. HTTP worker ignores any extra fields.
`run_id` is new — result consumer uses it to target the exact `job_runs` row.

**LLM dispatch (API result consumer → LLM worker):**
```json
{
  "job_id": "uuid",
  "run_id": "uuid",
  "raw_minio_path": "scrapeflow-results/history/uuid/1743516000.html",
  "provider": "openai_compatible|anthropic",
  "encrypted_api_key": "<fernet-ciphertext>",
  "base_url": "https://vllm.example.com/v1",
  "model": "Qwen/Qwen2.5-72b-Instruct",
  "output_schema": { ... }
}
```
`encrypted_api_key` is the Fernet ciphertext from `user_llm_keys.encrypted_api_key`. LLM worker decrypts with `LLM_KEY_ENCRYPTION_KEY`.

**Job result (all workers → API):**
```json
{
  "job_id": "uuid",
  "run_id": "uuid",
  "status": "running|completed|failed",
  "minio_path": "scrapeflow-results/history/uuid/1743516000.json",
  "nats_stream_seq": 42,
  "error": "optional, only when status=failed"
}
```
`run_id` is new — required for result consumer to update the correct row.
`nats_stream_seq` is present **only on `status: "running"` messages**. Workers read this from `msg.Metadata().Sequence.Stream` and include it so the result consumer can store it on `job_runs.nats_stream_seq`. The MaxDeliver advisory subscriber uses this value to identify which run to mark failed (see §6.3).
`minio_path` stores the `history/` path (see §2 Issue 14 fix) — the run's immutable result. The `latest/` path is always derivable as `latest/{job_id}.{ext}`.

## 4. New Services

### 4.1 Go HTTP Worker — Changes Only

The Go HTTP worker requires minimal changes for Phase 2:

- **Subscription subject:** `scrapeflow.jobs.run` → `scrapeflow.jobs.run.http`
- **Message schema:** Accept `run_id` field, include in all published results
- **MinIO path:** Write to both `latest/{job_id}.{ext}` AND `history/{job_id}/{timestamp}.{ext}`
- **Structured logging:** Replace `log.Printf` with `slog.Info/Error` (key-value pairs)
- **Pull consumer:** Switch from push subscription to pull consumer with configurable worker pool

**Pull consumer pattern (Go):**
```go
sub, err := js.PullSubscribe(
    NATS_JOBS_RUN_HTTP_SUBJECT,
    "go-http-worker",
    nats.MaxDeliver(cfg.NATSMaxDeliver),
)

// Worker pool — semaphore pattern
sem := make(chan struct{}, cfg.WorkerPoolSize)  // default: runtime.NumCPU()

for {
    // Only fetch as many messages as there are available slots.
    // Fetching more than available capacity causes messages to wait in-process
    // while NATS's AckWait timer runs — leading to redelivery and duplicate processing.
    available := cap(sem) - len(sem)
    if available == 0 {
        time.Sleep(100 * time.Millisecond)
        continue
    }
    msgs, err := sub.Fetch(available, nats.MaxWait(5*time.Second))
    if err != nil {
        continue  // timeout or transient error — loop and retry
    }
    for _, msg := range msgs {
        sem <- struct{}{}  // always succeeds: we fetched exactly `available` messages
        go func(m *nats.Msg) {
            defer func() { <-sem }()
            w.handleMessage(ctx, m)
        }(msg)
    }
}
```

### 4.2 Python Playwright Worker — New Service

**Location:** `playwright-worker/`
**Base image:** `mcr.microsoft.com/playwright/python:v1.44.0-jammy`

**Startup sequence:**
1. Load config from env vars
2. Connect to NATS, verify SCRAPEFLOW stream exists
3. Connect to MinIO, verify bucket exists
4. Launch Chromium browser (`playwright.chromium.launch(headless=True)`)
5. Create pull consumer on `scrapeflow.jobs.run.playwright`
6. Run worker loop (concurrency: `PLAYWRIGHT_MAX_WORKERS`, default 3)

**Per-job lifecycle:**
```python
async def process_job(msg, job):
    # 1. Publish running status
    await publish_result({"job_id": job.job_id, "run_id": job.run_id, "status": "running"})

    # 2. Create isolated browser context
    context = await browser.new_context()
    page = await context.new_page()

    try:
        # 3. Optional resource blocking
        if job.playwright_options.block_images:
            await page.route("**/*.{png,jpg,jpeg,gif,webp,svg,woff,woff2,ttf,css}",
                           lambda r: r.abort())

        # 4. Navigate
        await page.goto(job.url, timeout=job.playwright_options.timeout_seconds * 1000)
        await page.wait_for_load_state(job.playwright_options.wait_strategy)

        # 5. Extract rendered HTML
        html = await page.content()
        final_url = page.url

        # 6. Format + upload (same as Go worker)
        result = format(html, job.output_format, final_url)
        minio_path = await upload(job.job_id, result)

        # 7. Publish completed
        await publish_result({
            "job_id": job.job_id, "run_id": job.run_id,
            "status": "completed", "minio_path": minio_path
        })
        await msg.ack()

    except Exception as e:
        await publish_result({
            "job_id": job.job_id, "run_id": job.run_id,
            "status": "failed", "error": str(e)
        })
        await msg.ack()

    finally:
        await context.close()   # always discard session state
```

### 4.3 Python LLM Worker — New Service

**Location:** `llm-worker/`
**Base image:** `python:3.12-slim`
**Dependencies:** `openai`, `anthropic`, `nats-py`, `miniopy-async`, `cryptography`, `structlog`

**Startup sequence:**
1. Load config (requires `LLM_KEY_ENCRYPTION_KEY`)
2. Connect to NATS, MinIO
3. Pull subscribe to `scrapeflow.jobs.llm` (durable: `python-llm-worker`)

**Per-job lifecycle:**
```python
async def process_job(msg, job):
    # 1. Decrypt the LLM API key
    fernet = Fernet(settings.llm_key_encryption_key)
    api_key = fernet.decrypt(job.encrypted_api_key.encode()).decode()

    # 2. Fetch raw content from MinIO
    raw_bytes = await minio.get_object(BUCKET, job.raw_minio_path)
    content = raw_bytes.decode("utf-8", errors="replace")

    # 3. Truncate if content exceeds context window budget
    # JS-heavy SPAs can produce 500KB+ of HTML — exceeds 128K token limits on all providers.
    # Truncation is preferable to a silent HTTP 400 from the LLM API with a cryptic
    # "context_length_exceeded" error that the user can't diagnose.
    if len(content) > settings.llm_max_content_chars:
        log.warning("Content truncated before LLM call",
                    job_id=job.job_id, original_chars=len(content),
                    truncated_to=settings.llm_max_content_chars)
        content = content[:settings.llm_max_content_chars]

    # 4. Call LLM
    if job.provider == "anthropic":
        result = await call_anthropic(api_key, content, job.model, job.output_schema)
    else:
        result = await call_openai_compatible(
            api_key, job.base_url, content, job.model, job.output_schema
        )

    # 5. Upload structured result to MinIO
    minio_path = await upload_json(job.job_id, result)

    # 6. Publish final completed result
    await publish_result({
        "job_id": job.job_id, "run_id": job.run_id,
        "status": "completed", "minio_path": minio_path
    })
    await msg.ack()
    # api_key goes out of scope here — not persisted anywhere
```

**LLM call — openai_compatible (covers OpenAI, vLLM, Groq, Together, Ollama):**
```python
timeout = httpx.Timeout(
    connect=5.0,
    read=settings.llm_request_timeout_seconds,   # default 120s
    write=10.0,
    pool=5.0,
)
client = AsyncOpenAI(api_key=api_key, base_url=base_url or None, timeout=timeout)
response = await client.chat.completions.create(
    model=model,
    messages=[{"role": "user", "content": f"Extract data from:\n\n{content}"}],
    response_format={
        "type": "json_schema",
        "json_schema": {"name": "extraction", "schema": output_schema}
    }
)
return json.loads(response.choices[0].message.content)
```

**LLM call — anthropic:**
```python
client = anthropic.AsyncAnthropic(
    api_key=api_key,
    timeout=httpx.Timeout(
        connect=5.0,
        read=settings.llm_request_timeout_seconds,
        write=10.0,
        pool=5.0,
    ),
)
response = await client.messages.create(
    model=model,
    max_tokens=4096,
    tools=[{"name": "extract", "input_schema": output_schema}],
    tool_choice={"type": "tool", "name": "extract"},
    messages=[{"role": "user", "content": f"Extract data from:\n\n{content}"}],

return response.content[0].input  # dict matching output_schema
```

### 5.3 Modified: `POST /jobs`

**New Pydantic models — define in `api/app/schemas/jobs.py`:**
```python
import enum
from pydantic import BaseModel, Field

class Engine(str, enum.Enum):
    http       = "http"
    playwright = "playwright"

class WaitStrategy(str, enum.Enum):
    load             = "load"              # default — waits for load event
    domcontentloaded = "domcontentloaded"  # faster, fires before images/scripts
    networkidle      = "networkidle"       # opt-in — waits for no network activity (slow)

class PlaywrightOptions(BaseModel):
    wait_strategy:   WaitStrategy = WaitStrategy.load
    timeout_seconds: int          = Field(default=60, ge=5, le=300)
    block_images:    bool         = False

class LLMJobConfig(BaseModel):
    llm_key_id:    str       # UUID reference to user_llm_keys.id
    model:         str       # e.g. "gpt-4o", "claude-3-5-sonnet-20241022"
    output_schema: dict      # JSON Schema object — passed to LLM as structured output constraint
```

`WaitStrategy` values map directly to Playwright's `page.wait_for_load_state()` API — any value not in this enum is rejected at `POST /jobs` with a 422, before it ever reaches the Playwright worker. `timeout_seconds` is bounded (5s min, 300s max) to prevent accidental infinite waits or unreasonably short timeouts.

**New request fields:**
```python
class JobCreate(BaseModel):
    url: HttpUrl
    output_format: OutputFormat = OutputFormat.html
    engine: Engine = Engine.http                   # new
    playwright_options: PlaywrightOptions | None = None  # new
    schedule_cron: str | None = None               # new
    webhook_url: HttpUrl | None = None             # new
    llm_config: LLMJobConfig | None = None         # new
```

**New validations at creation:**
1. SSRF check on `url` (existing)
2. SSRF check on `webhook_url` (if provided)
3. SSRF check on LLM key's `base_url` (resolved from `llm_key_id`)
4. Validate `schedule_cron` with `croniter`
5. Enforce `SCHEDULE_MIN_INTERVAL_MINUTES` — return 422 if too frequent
6. Validate `llm_key_id` exists and belongs to user — return 404 if not found

**New creation flow:**
```
1. If webhook_url set: generate webhook_secret, Fernet-encrypt → hold in memory
2. Create jobs row (with webhook_secret already set if applicable)
3. Create job_runs row (status=pending)
4. Commit DB transaction
5. Publish to NATS with {job_id, run_id, url, output_format, engine, playwright_options}
6. Return JobResponse with job_id + run_id + webhook_secret (shown once if generated)
```

**Rationale:** The webhook secret must exist in the DB before the job enters the NATS queue. If step 5 fires before step 2 commits (possible on fast local networks), the webhook delivery loop would attempt to read `job.webhook_secret` before it exists — burning a retry slot before the job's first delivery. Generating the secret as part of the initial INSERT eliminates this race entirely.

## 6. Background Tasks

Three new background tasks, all started in the FastAPI lifespan context manager after existing client startup.

### 6.1 Scheduler Loop

**Ordering rule:** DB commit happens **before** NATS publish. This follows ADR-001's principle that the database is the source of truth and NATS is the delivery mechanism. If NATS publish fails after a successful commit, the `job_runs` row remains as `pending` and the stale-pending recovery path (below) re-dispatches it on the next cycle. The inverse — NATS has an in-flight message but DB has no row — has no recovery path.

```python
async def scheduler_loop(db_factory, js):
    while True:
        try:
            await asyncio.sleep(60)
            async with db_factory() as db:
                # FOR UPDATE SKIP LOCKED prevents double-dispatch in multi-instance deploy
                due = await db.execute(
                    SELECT jobs
                    WHERE schedule_cron IS NOT NULL
                      AND schedule_status = 'active'
                      AND next_run_at <= NOW()
                    FOR UPDATE SKIP LOCKED
                )

                created_runs = []
                for job in due:
                    run = await create_job_run(db, job.id, status='pending')
                    next_run = croniter(job.schedule_cron, datetime.now(UTC)).get_next(datetime)
                    await db.execute(
                        UPDATE jobs SET next_run_at = next_run, last_run_at = NOW()
                        WHERE id = job.id
                    )
                    created_runs.append((job, run))

                await db.commit()  # commit ALL runs and next_run_at updates before touching NATS

            # Publish after commit — outside the DB transaction
            for job, run in created_runs:
                try:
                    await dispatch_to_nats(js, job, run.id)
                except Exception:
                    logger.exception("NATS dispatch failed", job_id=str(job.id), run_id=str(run.id))
                    # job_run stays 'pending' — stale-pending recovery will re-dispatch

        except asyncio.CancelledError:
            raise
        except Exception:
            logger.exception("Scheduler loop error — will retry next cycle")
```

**Stale-pending recovery:** On each scheduler cycle, after processing due jobs, also re-dispatch any `job_runs` where `status = 'pending'` AND `created_at < NOW() - interval '10 minutes'`. These are runs where the DB commit succeeded but NATS publish failed.

```python
stale = await db.execute(
    SELECT jr.*, j.* FROM job_runs jr
    JOIN jobs j ON jr.job_id = j.id
    WHERE jr.status = 'pending'
      AND jr.created_at < NOW() - interval '10 minutes'
    FOR UPDATE SKIP LOCKED
)
for job, run in stale:
    await dispatch_to_nats(js, job, run.id)
```

### 6.2 Webhook Delivery Loop

```python
BACKOFF_SECONDS = [0, 30, 300, 1800, 7200]  # attempts 1–5

async def webhook_delivery_loop(db_factory, http_client, fernet):
    while True:
        await asyncio.sleep(15)
        async with db_factory() as db:
            pending = await db.execute(
                SELECT webhook_deliveries
                WHERE status = 'pending' AND next_attempt_at <= NOW()
                FOR UPDATE SKIP LOCKED
                LIMIT 50
            )
            for delivery in pending:
                await attempt_delivery(db, http_client, fernet, delivery)

async def attempt_delivery(db, http_client, fernet, delivery):
    # Known gap — SSRF not re-validated at delivery time (Phase 3 hardening item)
    # DNS rebinding attacks can change what IP a hostname resolves to after the initial
    # SSRF check at POST /jobs time. At delivery, we POST to delivery.webhook_url without
    # re-resolving. Fix: call _validate_no_ssrf(delivery.webhook_url) here before the POST.
    # Deferred because: (a) project is dev-only, (b) _validate_no_ssrf() is being moved to
    # core/security.py in §5.1 — the infrastructure for this fix is already planned.
    # Add to Phase 3 backlog: re-validate webhook_url on every delivery attempt.
    job = await db.get(Job, delivery.job_id)
    secret = fernet.decrypt(job.webhook_secret.encode()).decode()

    payload_bytes = json.dumps(delivery.payload).encode()
    sig = hmac.new(secret.encode(), payload_bytes, hashlib.sha256).hexdigest()

    try:
        resp = await http_client.post(
            delivery.webhook_url,
            content=payload_bytes,
            headers={
                "Content-Type": "application/json",
                "X-ScrapeFlow-Signature": f"sha256={sig}",
            },
            timeout=10.0,
        )
        if resp.status_code < 300:
            await db.execute(
                UPDATE webhook_deliveries
                SET status='delivered', delivered_at=NOW()
                WHERE id = delivery.id
            )
            return
        error = f"HTTP {resp.status_code}"
    except Exception as e:
        error = str(e)

    attempts = delivery.attempts + 1
    if attempts >= settings.webhook_max_attempts:
        new_status = "exhausted"
        next_attempt = None
    else:
        new_status = "pending"
        next_attempt = datetime.now(UTC) + timedelta(seconds=BACKOFF_SECONDS[attempts])

    await db.execute(
        UPDATE webhook_deliveries
        SET attempts=attempts, status=new_status,
            next_attempt_at=next_attempt, last_error=error
        WHERE id = delivery.id
    )
```

### 6.3 MaxDeliver Advisory Subscriber

When NATS exhausts redeliveries for a message, it publishes to `$JS.EVENT.ADVISORY.CONSUMER.MAX_DELIVERIES.SCRAPEFLOW.*`.

```python
async def maxdeliver_advisory_subscriber(nats_client, db_factory):
    await nats_client.subscribe(
        "$JS.EVENT.ADVISORY.CONSUMER.MAX_DELIVERIES.SCRAPEFLOW.*",
        cb=lambda msg: asyncio.create_task(handle_maxdeliver_advisory(msg, db_factory)),
    )

async def handle_maxdeliver_advisory(msg, db_factory):
    advisory = json.loads(msg.data)
    stream_seq = advisory["stream_seq"]

    # The advisory contains only metadata — no job_id/run_id.
    # job_runs.nats_stream_seq was stored when the result consumer first received
    # the 'running' status message from the worker (see §3.3, §7).
    async with db_factory() as db:
        await db.execute("""
            UPDATE job_runs
            SET status = 'failed',
                error  = 'Max NATS redeliveries exceeded',
                completed_at = NOW()
            WHERE nats_stream_seq = :seq
              AND status IN ('pending', 'running', 'processing')
        """, {"seq": stream_seq})
        # 'processing' is included: LLM jobs can exhaust redeliveries on the
        # scrapeflow.jobs.llm subject while in processing state (Issue 9 fix)
```

**`nats_stream_seq` lifecycle:**
- Added in migration 2.8 — `BIGINT NULL` on `job_runs`
- Set by: the result consumer, on receipt of the first `status: "running"` message
- Value source: worker reads `msg.Metadata().Sequence.Stream` and includes it as `nats_stream_seq` in the result payload
- Used by: this advisory handler to identify and fail the stalled run

### 6.4 Lifespan Integration

Each background task is launched with `asyncio.create_task()` — **not** `asyncio.TaskGroup`. TaskGroup blocks until all tasks complete; since every loop is `while True`, it would prevent FastAPI from ever reaching `yield` and the API would hang on startup.

Each loop's body must catch all non-cancellation exceptions internally and log + continue. Without this, a transient DB error permanently kills the task with no visible failure. `asyncio.CancelledError` must be re-raised so the shutdown signal propagates correctly.

```python
# api/app/main.py — lifespan (startup)
scheduler_task = asyncio.create_task(scheduler_loop(get_db, app.state.nats_js))
webhook_task   = asyncio.create_task(webhook_delivery_loop(get_db, httpx_client, fernet))
result_task    = asyncio.create_task(start_result_consumer(app.state.nats_js))
advisory_task  = asyncio.create_task(
    maxdeliver_advisory_subscriber(app.state.nats_client, get_db)
)

yield  # FastAPI serves here

# Shutdown — cancel in reverse start order
for task in [advisory_task, scheduler_task, webhook_task, result_task]:
    task.cancel()
    await asyncio.gather(task, return_exceptions=True)
```

**Required pattern for every background loop:**

```python
async def scheduler_loop(db_factory, js):
    while True:
        try:
            await asyncio.sleep(60)
            async with db_factory() as db:
                ...
        except asyncio.CancelledError:
            raise   # propagate shutdown signal — do not swallow
        except Exception:
            logger.exception("Scheduler loop error — will retry next cycle")
```

Apply this same `try/except` structure to `webhook_delivery_loop` and `maxdeliver_advisory_subscriber`.

## 7. Result Consumer Changes

The result consumer (`api/app/core/result_consumer.py`) gains two new responsibilities:

**Cancellation guard — runs at the top of every result message handler:**
```python
# ADR-001 cancellation enforcement — Phase 2 uses run_id (jobs.status was removed in migration 2.4)
run = await db.get(JobRun, run_id)
if run is None:
    await msg.ack()
    return  # stale or deleted job — discard silently

if run.status == 'cancelled':
    await msg.ack()
    return  # job was cancelled before worker completed — discard result
```

This replaces the Phase 1 `if job.status == JobStatus.cancelled` check. Since Phase 2 messages carry `run_id`, we look up the specific run directly. The `job.status` column no longer exists after migration 2.4.

---

**Discriminator invariant — how the consumer distinguishes scrape results from LLM results:**

Both scrape workers (Go HTTP, Python Playwright) and the Python LLM worker publish to `scrapeflow.jobs.result` with identical message schemas. There is no `worker_type` field. The consumer uses `job_run.status` as the discriminator — this works because only the result consumer transitions the run to `processing`, and it does so only when dispatching to the LLM subject:

| `job_run.status` when `completed` arrives | Source | Action |
|-------------------------------------------|--------|--------|
| `running` | Scrape worker | Check `llm_config` — route to LLM if set, finalize if not |
| `processing` | LLM worker | Skip LLM routing — diff and finalize |

**This invariant must be preserved.** No code path other than the LLM dispatch branch in the scrape-complete handler may set `job_run.status = 'processing'`.

---

**On `status: "completed"` from a scrape worker:**
```python
# 1. Update job_run — store the history/ path as result_path
await db.execute(UPDATE job_runs SET status='completed', result_path=minio_path,
                  completed_at=NOW() WHERE id = run_id)

# 2. Check if LLM processing needed — BEFORE computing diff
job = await db.get(Job, job_id)
if job.llm_config:
    # Diff is deferred: for LLM jobs, diff runs after LLM extraction so we compare
    # structured JSON outputs, not raw HTML. Computing diff here would compare raw
    # HTML against raw HTML from the previous run — defeating the purpose entirely.
    await db.execute(UPDATE job_runs SET status='processing' WHERE id = run_id)
    llm_key = await db.get(UserLLMKey, job.llm_config['llm_key_id'])
    if llm_key is None:
        await db.execute(UPDATE job_runs SET status='failed',
                          error='LLM key not found or deleted' WHERE id = run_id)
    else:
        await js.publish(NATS_JOBS_LLM_SUBJECT, build_llm_message(job, run_id, llm_key, minio_path))
        return  # do NOT diff or create webhook delivery yet — wait for LLM completion

# 3. Compute diff (non-LLM jobs only — normalised text diff on scraped content)
prev_run = await get_previous_completed_run(db, job_id)
if prev_run:
    diff = await compute_text_diff(minio_path, prev_run.result_path)
    await db.execute(UPDATE job_runs SET diff_detected=diff.detected,
                      diff_summary=diff.summary WHERE id = run_id)

# 4. Create webhook delivery if applicable
if job.webhook_url:
    await create_webhook_delivery(db, job, run_id, event="job.completed", minio_path=minio_path)
```

**On `status: "completed"` from the LLM worker:**
```python

await db.execute(UPDATE job_runs SET status='completed', result_path=minio_path,
                  completed_at=NOW() WHERE id = run_id)

# 2. Compute JSON diff (LLM jobs — field-by-field comparison of structured output)

    diff = await compute_json_diff(minio_path, prev_run.result_path)

# 3. Create webhook delivery

if job.webhook_url:
    await create_webhook_delivery(db, job, run_id, event="job.completed", minio_path=minio_path)
```

**Webhook event design — chosen approach and alternatives:**

### Option A — Single event with diff embedded (implemented)

One `job.completed` event per run. The payload always includes `diff_detected` and `diff_summary` (null on first run, populated on subsequent runs). Consumer inspects `diff_detected` to decide whether to act on the change.

**Webhook payload schema:**
```json
{
  "event":        "job.completed | job.failed",
  "job_id":       "uuid",
  "run_id":       "uuid",
  "result_path":  "scrapeflow-results/history/uuid/1743516000.json",
  "diff_detected": true,
  "diff_summary": {
    "price": {"from": 49.99, "to": 39.99},
    "in_stock": {"from": true, "to": false}
  },
  "error":        "only present when event=job.failed",
  "timestamp":    "2026-04-01T18:05:00Z"
}
```

`diff_detected` is `null` on the first run for a job (no previous run to compare against), `true`/`false` on subsequent runs. `diff_summary` is `null` when `diff_detected` is `false` or `null`.

**Why Option A:** Consistent with standard webhook platforms (Stripe, GitHub, Shopify) — one event per logical outcome, consumer inspects the payload. No need to register or handle multiple event types for full coverage. The payload contains everything the consumer needs to act.

---

### Option B — Separate `job.change_detected` event (not implemented)

Fire `job.completed` when a run completes without a change, and a separate `job.change_detected` event (instead of `job.completed`) when a diff is detected.

**Tradeoffs vs Option A:**
- **Pro:** Consumer can semantically filter — a change-monitoring pipeline subscribes only to `job.change_detected` and ignores completions entirely
- **Con:** Consumer needs handlers for two event types to get full run visibility
- **Con:** `job.completed` still fires for no-change runs, so a consumer tracking all completions needs both handlers regardless
- **Con:** Non-standard pattern — most webhook platforms (Stripe, GitHub) use one event + payload inspection, not event-type filtering
- **Reconsider if:** Phase 3 adds per-event webhook subscriptions where users configure which events to receive — at that point Option B's semantic filtering becomes a real UX win

---

**Webhook delivery creation (both handlers):**

```python
# Create single delivery — diff is embedded in payload
if job.webhook_url:
    await create_webhook_delivery(db, job, run_id, event="job.completed",
                                   minio_path=minio_path, diff=diff)
```

No second delivery for change detection. The `diff_detected` and `diff_summary` fields in the payload carry the change information.

### 8.3 `scripts/cleanup_old_runs.py` — Run History Cleanup

Runs nightly via the k3s CronJob (§8.2). Deletes `job_runs` older than `SCHEDULE_RUN_RETENTION_DAYS` (default 90), their associated `webhook_deliveries` rows, and their `history/` MinIO objects.

**Deletion order — MinIO first, DB second:**

MinIO and Postgres are separate systems with no shared transaction. If we delete the DB row first and the MinIO delete fails, the object is orphaned in MinIO forever with no reference to find it. If we delete MinIO first and the DB delete fails, the DB row still exists — the next nightly run will retry and find the MinIO object already gone (idempotent). Always delete external state before internal state.

```python
import os, asyncio, structlog
from datetime import datetime, timedelta, UTC
from miniopy_async import Minio
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession

log = structlog.get_logger()

RETENTION_DAYS = int(os.environ["SCHEDULE_RUN_RETENTION_DAYS"])
BUCKET = os.environ["MINIO_BUCKET"]
cutoff = datetime.now(UTC) - timedelta(days=RETENTION_DAYS)

async def main():
    engine = create_async_engine(os.environ["DATABASE_URL"])
    minio  = Minio(os.environ["MINIO_ENDPOINT"], ...)

    async with AsyncSession(engine) as db:
        # 1. Fetch expired runs in batches (avoid loading millions of rows at once)
        while True:
            rows = await db.execute("""
                SELECT id, job_id, result_path
                FROM job_runs
                WHERE created_at < :cutoff
                ORDER BY created_at
                LIMIT 500
            """, {"cutoff": cutoff})
            runs = rows.fetchall()
            if not runs:
                break

            # 2. Delete MinIO history/ objects first
            for run in runs:
                if run.result_path and run.result_path.startswith("history/"):
                    try:
                        await minio.remove_object(BUCKET, run.result_path)
                    except Exception:
                        log.exception("MinIO delete failed — skipping DB delete for this run",
                                      run_id=str(run.id), path=run.result_path)
                        continue  # leave DB row intact — will retry next night

            # 3. Delete webhook_deliveries rows (CASCADE would handle this, but explicit
            #    delete avoids long-running CASCADE on large datasets)
            run_ids = [r.id for r in runs]
            await db.execute("""
                DELETE FROM webhook_deliveries WHERE run_id = ANY(:ids)
            """, {"ids": run_ids})

            # 4. Delete job_runs rows
            await db.execute("""
                DELETE FROM job_runs WHERE id = ANY(:ids) AND created_at < :cutoff
            """, {"ids": run_ids, "cutoff": cutoff})

            await db.commit()
            log.info("Cleanup batch complete", deleted=len(runs))

asyncio.run(main())
```

**Scope:**
- Deletes: `job_runs.created_at < cutoff`, their `webhook_deliveries`, their `history/{job_id}/{timestamp}.*` MinIO objects
- Never deletes: `latest/{job_id}.*` MinIO objects — these always reflect current scraped state
- Never deletes: `jobs` template rows — those are deleted only by user/admin via `DELETE /jobs/{id}`

**Idempotency:** Safe to re-run. If a batch fails mid-way, the next run picks up where it left off (rows not yet deleted still match the `created_at < cutoff` query). MinIO deletes that already succeeded are silently skipped by MinIO (returns 204 on a non-existent object).

## 9. Environment Variables Reference

| Variable | Service | Default | Description |
|----------|---------|---------|-------------|
| `LLM_KEY_ENCRYPTION_KEY` | API, LLM worker | required | Fernet key for LLM API keys + webhook secrets |

| `WEBHOOK_MAX_ATTEMPTS` | API | `5` | Max webhook delivery attempts before exhausted |
| `PLAYWRIGHT_MAX_WORKERS` | Playwright worker | `3` | Concurrent browser pages |
| `PLAYWRIGHT_DEFAULT_TIMEOUT_SECONDS` | Playwright worker | `60` | Default page load timeout |

## 10. Testing Requirements

### 10.1 General Rules
- All integration tests run via `docker compose exec api python -m pytest`
- Real Postgres, Redis, NATS, MinIO — no mocking infrastructure
- Mock: Clerk (already done), external LLM APIs, external webhook endpoints

### 10.2 Required Test Coverage

**Migrations:**
- Run full migration suite on empty DB, verify all tables and constraints
- Run data migration (2.3) against a seeded DB with existing jobs, verify job_runs populated correctly

**LLM Keys:**
- Create key → verify stored encrypted (raw key not visible in DB)
- List keys → verify api_key masked
- Delete key → verify hard deleted
- Create job referencing deleted key → verify 404 at job creation

**Playwright Worker:**
- Mock Chromium: `playwright.chromium.launch()` with a mock browser
- Verify `networkidle` and `load` wait strategies both dispatch correctly
- Verify context is closed after every job (success and failure)

**Scheduler:**
- Seed a job with `next_run_at = NOW() - 1 minute`, run one loop iteration
- Verify `job_runs` row created, NATS message published, `next_run_at` updated
- Seed two API instances (two scheduler loops), verify job dispatched exactly once

**Webhook Delivery:**
- Mock httpx: return 2xx → verify `status=delivered`
- Mock httpx: return 500 → verify `attempts++`, `next_attempt_at` uses backoff
- After 5 failures → verify `status=exhausted`
- Verify HMAC header present and correct on every delivery attempt
- Admin retry endpoint → verify `status=pending`, `attempts=0`

**Admin Panel:**
- Every `/admin/*` route: assert 403 for non-admin user
- Every `/admin/*` route: assert 200 for admin user with correct data
- Stats endpoint: seed known data, verify counts match

**Change Detection:**
- First run: verify `diff_detected=NULL`
- Second run, same content: verify `diff_detected=false`
- Second run, changed content: verify `diff_detected=true`, `diff_summary` populated
- LLM job: verify field-by-field JSON diff in `diff_summary`

**MaxDeliver Advisory:**
- Simulate NATS advisory message, verify `job_runs.status` set to `failed`
