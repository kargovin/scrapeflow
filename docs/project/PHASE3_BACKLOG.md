## Dependency Groups and Sequencing

Phase 3 has three hard-ordering constraints:

1. **Workers before API (ADR-004):** Steps 14–15 (worker schema_version 2 updates) must be deployed to production **before** any API step that sends v2 fat messages. Steps 16–20 depend on this.
2. **Migration 3.6 is a shared gate:** Batch scraping (Step 21) and content deduplication (Step 27) both require migration 3.6 (`job_runs` nullable `job_id` + `batch_item_id` + `content_hash`). That migration depends on the `batch_items` table existing (migration 3.5, Step 8).
3. **Migration 3.4 before Steps 17–18:** The `job_secrets` table (migration 3.4, hand-written ENUM) is required by both proxy rotation and authenticated scraping. It cannot be autogenerated — use the `COMMIT`/`BEGIN` pattern.

#### Step 2 — Sliding window rate limiter (PRD-002)

**Why this step:** The current fixed-window counter has a 2× burst exploit (request at end of window N + start of window N+1 = 2× quota). Replace with a Redis sorted-set + Lua atomic sliding window.

**Location:** `api/app/core/rate_limit.py` — replace `check_rate_limit` entirely

**What to build:**

- Replace the `INCR`/`EXPIRE` implementation with a Lua script (`ZREMRANGEBYSCORE` → `ZCARD` → conditional `ZADD` + `EXPIRE`)
- Function signature stays the same — no changes to callers (`POST /jobs` dependency)
- Lua script atomicity ensures no concurrent-request exploit (two requests cannot both read `count < limit`)
- Key format: `rate:user:<user_id>` (unchanged)
- Window: `RATE_LIMIT_WINDOW_SECONDS` env var, default 60 (previously hardcoded — add to `settings.py`)
- Member uniqueness: `now_ms .. '-' .. random()` prevents collision when two requests arrive in the same millisecond

**Tests to write/update:**
- Under limit: allowed
- At limit: allowed (boundary is inclusive)
- Over limit: 429
- Two concurrent requests both at limit: only one allowed (atomicity test)
- Window expiry: counter resets after window passes
- Existing test file: `tests/test_rate_limit.py`

**No migration needed.** Drop-in replacement.

#### Step 3 — SSRF re-validation on webhook delivery (PRD-003)

**Why this step:** DNS rebinding attack — a webhook URL can resolve to a public IP at `POST /jobs` time, then rebind to an internal IP (`10.x.x.x`, `169.254.x.x`) before delivery. The Phase 2 check only runs at job creation.

**Location:** `api/app/services/webhook_delivery.py` — `_attempt_delivery()` function

**What to build:**

- Call `_validate_no_ssrf(delivery.webhook_url)` at the top of `_attempt_delivery()`
- On `ValueError` from SSRF check: call `_mark_exhausted(delivery, error=f"ssrf_blocked: {e}", session=session)` — no retry. SSRF-blocked deliveries are `exhausted`, not `failed`.
- Log a structured warning with `job_id`, `delivery_id`, and the blocked URL
- Do NOT retry SSRF-blocked deliveries — retries probe internal addresses repeatedly

**Tests to write:**
- Delivery to a public URL succeeds (mock SSRF validator allows)
- Delivery to a rebinding URL gets marked `exhausted` immediately (no retry)
- `exhausted` status is set on the delivery row, not `failed`

#### Step 7 — Migration 3.4: `job_secrets` table + `job_secret_type` ENUM (ADR-007) ⚠ HAND-WRITTEN

**Location:** New Alembic migration — must be hand-written; autogenerate cannot create ENUMs

**Why hand-written:** Creating a PostgreSQL ENUM inside a transaction requires the `COMMIT`/`BEGIN` pattern established in Phase 2 for `jobstatus`. The ENUM `job_secret_type` must be created before the `job_secrets` table references it.

**What to build:**

```python
def upgrade() -> None:
    # ENUM creation must run outside a transaction
    op.execute(sa.text("COMMIT"))
    op.execute(sa.text("CREATE TYPE job_secret_type AS ENUM ('proxy', 'cookies')"))
    op.execute(sa.text("BEGIN"))

    op.create_table(
        'job_secrets',
        sa.Column('id', sa.UUID(), primary_key=True, server_default=sa.text('gen_random_uuid()')),
        sa.Column('job_id', sa.UUID(), sa.ForeignKey('jobs.id', ondelete='CASCADE'), nullable=False),
        sa.Column('secret_type', sa.Enum('proxy', 'cookies', name='job_secret_type'), nullable=False),
        sa.Column('encrypted_value', sa.Text(), nullable=False),
        sa.Column('created_at', sa.TIMESTAMPTZ(), server_default=sa.text('NOW()'), nullable=False),
        sa.Column('updated_at', sa.TIMESTAMPTZ(), server_default=sa.text('NOW()'), nullable=False),
        sa.UniqueConstraint('job_id', 'secret_type', name='uq_job_secrets_job_type'),
    )
    op.create_index('idx_job_secrets_job_id', 'job_secrets', ['job_id'])

def downgrade() -> None:
    op.drop_index('idx_job_secrets_job_id', table_name='job_secrets')
    op.drop_table('job_secrets')
    op.execute(sa.text("COMMIT"))
    op.execute(sa.text("DROP TYPE IF EXISTS job_secret_type"))
    op.execute(sa.text("BEGIN"))
```

Add the `JobSecret` SQLAlchemy model to `api/app/models/` and the `JobSecretType` Python Enum to match.

#### Step 9 — Migration 3.6: `job_runs` — nullable `job_id`, `batch_item_id`, check constraint, `content_hash` (ADR-006, PRD-015) ⚠ HAND-WRITTEN CONSTRAINT

**Why this step is a gate:** Batch scraping (Step 21) and content dedup (Step 27) both depend on this migration. This is the riskiest migration in Phase 3 — it touches a high-traffic table and drops a NOT NULL constraint.

**Location:** New Alembic migration — autogenerate for column changes + manual constraint append

**What to build:**

```python
def upgrade() -> None:
    # Drop NOT NULL on job_id (autogenerate handles this)
    op.alter_column('job_runs', 'job_id', nullable=True)

    # Add batch_item_id FK (autogenerate handles this after model update)
    op.add_column('job_runs', sa.Column('batch_item_id', sa.UUID(), nullable=True))
    op.create_foreign_key(
        'fk_job_runs_batch_item_id',
        'job_runs', 'batch_items',
        ['batch_item_id'], ['id']
    )

    # Add content_hash for dedup (PRD-015)
    op.add_column('job_runs', sa.Column('content_hash', sa.String(16), nullable=True))

    # Mutual exclusion constraint + partial index (MUST be hand-written — autogenerate skips CHECK constraints)
    op.execute(sa.text("""
        ALTER TABLE job_runs
            ADD CONSTRAINT chk_job_runs_single_parent
                CHECK ((job_id IS NOT NULL) != (batch_item_id IS NOT NULL))
    """))
    op.create_index(
        'idx_job_runs_batch_item_id',
        'job_runs',
        ['batch_item_id'],
        postgresql_where=sa.text('batch_item_id IS NOT NULL')
    )
```

**Pre-migration check:** `SELECT COUNT(*) FROM job_runs WHERE job_id IS NULL` — must return 0 before running. If it doesn't, stop and investigate.

**After migration:** Audit all queries in the codebase that do `JOIN jobs ON job_runs.job_id = jobs.id` — change to `LEFT JOIN` or add `WHERE job_runs.job_id IS NOT NULL` as appropriate.

#### Step 12 — Migration 3.9: `jobs.updated_at` DB trigger (spec §2.3) ⚠ HAND-WRITTEN

**Why this step:** Mutation paths that use `db.execute(update(...))` bypass SQLAlchemy's `onupdate` hook silently. A Postgres trigger fires on every UPDATE regardless of path.

**Location:** New Alembic migration — hand-written DDL

**What to build:**

```python
def upgrade() -> None:
    op.execute(sa.text("""
        CREATE OR REPLACE FUNCTION set_jobs_updated_at()
        RETURNS TRIGGER AS $$
        BEGIN
            NEW.updated_at = NOW();
            RETURN NEW;
        END;
        $$ LANGUAGE plpgsql;

        CREATE TRIGGER trg_jobs_updated_at
        BEFORE UPDATE ON jobs
        FOR EACH ROW EXECUTE FUNCTION set_jobs_updated_at();
    """))

def downgrade() -> None:
    op.execute(sa.text("""
        DROP TRIGGER IF EXISTS trg_jobs_updated_at ON jobs;
        DROP FUNCTION IF EXISTS set_jobs_updated_at();
    """))
```

**After migration:** Remove `onupdate=lambda: datetime.now(UTC)` from `jobs.updated_at` in the SQLAlchemy model. The trigger is now authoritative.

2. **`POST /batch` logic:**
   - Accept `{ urls: list[str], output_format, engine, webhook_url, respect_robots }` (1–500 URLs)
   - Validate and SSRF-check all URLs synchronously (up to 500 — CPU-bound, 2–3 seconds; sync rejection is better UX than silent per-item failure)
   - Rate limit check: deduct `len(urls)` quota units upfront
   - Create `batches` row with `total = len(urls)`, `status = 'queued'`
   - Create `batch_items` rows (one per URL)
   - For each item: create a `job_runs` row with `batch_item_id` set and `job_id = null`; dispatch NATS message with standard fat message schema (`run_id`, `url`, etc. — no `batch_item_id` in the message itself per ADR-006 §4)
   - Return `{ batch_id, total, status }`
   - Set `batches.status = 'running'` after first dispatch

3. **Result consumer batch branch** (in `result_consumer.py`):
   - After loading `job_runs` row: if `run.batch_item_id IS NOT NULL`, route to batch branch
   - Batch branch: update `batch_items` row (`status`, `result_path`, `error`, `completed_at`); atomic `UPDATE batches SET completed = completed + 1` (or `failed = failed + 1`); if `completed + failed == total`: set `batches.status = 'completed'` or `'partial_failure'`, fire batch.completed webhook (respects `webhook_events` filter from Step 20)

4. **`DELETE /batch/{id}`:** Set all `pending`/`running` `batch_items` to `cancelled`; result consumer already discards results for cancelled runs via status check

**Tests:**
- `POST /batch` with 3 URLs → 3 `batch_items`, 3 NATS messages dispatched
- `POST /batch` with SSRF URL → 422 before any rows created
- Worker completes all 3 items → `batches.status = 'completed'`
- Worker fails one item → `batches.status = 'partial_failure'`
- `DELETE /batch/{id}` → items cancelled
- `GET /batch/{id}` returns accurate counters at each state transition
- `GET /jobs` does not include batch runs (filter: `WHERE job_runs.job_id IS NOT NULL` or query `jobs` table directly)

#### Step 23 — PRD-007: coordinator service

**Depends on:** Steps 10, 22

**Location:** `coordinator/` (new monorepo service)

**What to build:**

```
coordinator/
  main.py           # entrypoint — startup + loops
  bfs.py            # dispatch loop + crawl completion detection
  result_handler.py # NATS result subscriber for crawl messages
  link_extractor.py # BeautifulSoup href extraction + filtering
  sitemap.py        # robots.txt sitemap discovery + XML parsing
  models.py         # imports shared SQLAlchemy models from api/app/models
  config.py         # env var config (DATABASE_URL, NATS_URL, MINIO_URL, etc.)
  Dockerfile
  requirements.txt  # aiohttp, nats-py, sqlalchemy[asyncio], asyncpg, beautifulsoup4, lxml, aiofiles
```

**Key implementation details:**

1. **Startup sequence:**
   - Subscribe to `scrapeflow.jobs.result` with its own durable consumer group (not the same as the API result consumer — separate pull consumer)
   - On startup re-enqueue stalled items: `UPDATE crawl_queue SET status = 'pending' WHERE status = 'dispatched' AND dispatched_at < NOW() - INTERVAL '10 minutes'`
   - Start two concurrent async tasks: `dispatch_loop()` and `result_handler_loop()`

2. **`result_handler_loop()`:**
   - Pull messages from `scrapeflow.jobs.result`
   - If `crawl_context` is null → ack and skip (API result consumer handles it)
   - If `crawl_context` is non-null → process crawl result:
     - Update `crawl_pages` row
     - Update `crawl_queue` row to `completed`/`failed`
     - Fetch raw HTML from MinIO, parse `<a href>` tags via `link_extractor.py`
     - Insert new URLs into `crawl_queue` (UNIQUE constraint handles deduplication silently)
     - If `ignore_sitemap = false` and this is the seed URL result → also run `sitemap.py` discovery

3. **`dispatch_loop()`:**
   - Poll `crawl_queue WHERE status = 'pending'` in batches of `COORDINATOR_DISPATCH_BATCH_SIZE` (default 10)
   - Check crawl is still active (`crawls.status NOT IN ('cancelled', 'completed', 'failed')`)
   - For each item: create a `crawl_pages` row, update `crawl_queue` to `dispatched`, dispatch NATS message with `crawl_context` sub-object
   - Crawl completion check: `SELECT COUNT(*) FROM crawl_queue WHERE crawl_id = :id AND status = 'pending'` — when 0 AND all dispatched items are terminal, set `crawls.status = 'completed'`, fire webhook

4. **`link_extractor.py`:** Extract `<a href>` tags; apply same-origin filter, `include_paths` allowlist, `exclude_paths` blocklist, `max_depth`, `max_pages` limits

5. **`sitemap.py`:** On seed URL result, fetch `{origin}/robots.txt`, extract `Sitemap:` directives, fetch each XML, parse `<loc>` entries, insert into `crawl_queue` with `depth = 1`

6. **Docker Compose:** Add `coordinator` service to `docker-compose.yml`, depending on `api`, `postgres`, `nats`

7. **K8s manifest:** Add `coordinator` Deployment to infra repo (separate from Step 1 — Step 1 covers Phase 2 services only)

**Tests:**
- Coordinator processes a seed URL result and enqueues discovered links
- Duplicate URL in `crawl_queue` is silently ignored (UNIQUE constraint)
- Cancelled crawl — coordinator skips dispatch for all pending items
- Crawl completion detected when queue is empty
- Stalled `dispatched` items re-enqueued on startup

#### Step 25 — PRD-012: billing and per-user quotas

**Depends on:** Step 11 (migration 3.8 — `user_quotas`), Step 21 (batch quota deduction at submission)

**Location:** `api/app/core/quota.py` (new), `api/app/routers/admin.py`, enforcement in `POST /jobs`, `POST /batch`, scheduler, result consumer

**What to build:**

1. **`core/quota.py`:** `check_user_quota(user_id, db, quota_type)` — three checks:
   - `monthly_runs`: `SELECT COUNT(*) FROM job_runs WHERE user_id = :uid AND created_at >= first_of_month`
   - `concurrent_jobs`: `SELECT COUNT(*) FROM job_runs WHERE user_id = :uid AND status IN ('queued', 'running')`
   - `storage_bytes`: `SELECT storage_bytes_used FROM user_quotas WHERE user_id = :uid`
   - Limits from `user_quotas` row; fall back to env var defaults (`DEFAULT_QUOTA_MONTHLY_RUNS=500`, `DEFAULT_QUOTA_CONCURRENT_JOBS=5`, `DEFAULT_QUOTA_STORAGE_BYTES=5368709120`)

2. **Enforcement points:**
   - `POST /jobs`: check all three dimensions before creating the job
   - `POST /batch`: check `monthly_runs` — deduct `len(urls)` upfront; check `concurrent_jobs`
   - Scheduler dispatch loop: re-check `monthly_runs` and `concurrent_jobs` before dispatching; if exceeded, skip and log structured warning (do NOT fail the job template)
   - Result consumer: check `storage_bytes_used + result_size <= limit` before MinIO write; if exceeded, fail run with `error = "storage_quota_exceeded"`

3. **`PATCH /admin/users/{id}/quota`:** Accept `{ monthly_runs_limit, concurrent_jobs_limit, storage_bytes_limit }` (all nullable/optional). Upsert into `user_quotas`.

4. **`storage_bytes_used` maintenance:**
   - On each successful MinIO write in result consumer: `UPDATE user_quotas SET storage_bytes_used = storage_bytes_used + :size WHERE user_id = :uid`
   - On each `cleanup_old_runs.py` delete: decrement accordingly

5. **429 response format:**
```json
{
  "error": "quota_exceeded",
  "quota_type": "monthly_runs",
  "message": "Monthly run limit reached (500/500). Quota resets on 2026-05-01.",
  "resets_at": "2026-05-01T00:00:00Z"
}
```

**Tests:**
- User at monthly limit → `POST /jobs` returns 429 with correct format
- Admin sets custom limit → enforced
- Storage quota exceeded → result consumer fails run with correct error
- Scheduler skip when concurrent limit reached → no 429, just skip + log

#### Step 27 — PRD-015: content deduplication

**Depends on:** Step 9 (migration 3.6 adds `job_runs.content_hash`)

**Location:** `api/app/core/result_consumer.py`

**What to build:**

2. In result consumer, after receiving completed result and before diff/LLM dispatch:
```python
import xxhash

content_hash = xxhash.xxh64(normalised_content).hexdigest()[:16]
current_run.content_hash = content_hash

previous_run = await db.scalar(
    select(JobRun)
    .where(JobRun.job_id == job_id, JobRun.status == 'completed')
    .order_by(JobRun.created_at.desc())
    .offset(1)   # second-most-recent; current run is most recent
    .limit(1)
)
if previous_run and previous_run.content_hash == content_hash:
    current_run.change_detected = False
    # Update latest/ MinIO object (maintain the "latest" contract — still current content)
    await upload_latest(job_id, content, ext)
    # Skip: no history/ write, no diff, no LLM dispatch, no webhook
    return
```

4. **Only applies to regular job path** (not batch, not crawl) — batch and crawl have no previous-run concept in Phase 3.

**Tests:**
- Same content twice → second run `change_detected = False`, no new `history/` object, no LLM dispatch triggered
- Different content → `change_detected = True`, full diff + history write proceeds
- `latest/` is always updated regardless of hash match
